# Bysheen Cross-Border Product Agent System

多 Agent 协同的跨境商品内容生产系统，用于将 1688 等中文货源平台的商品信息自动转化为适合 TikTok Shop、Shopee、Lazada 等跨境电商平台上架的本地化商品内容。

本项目目标是通过 AI Agent 工作流，自动完成商品采集、内容清洗、多语言翻译、图片检测、图片优化、平台合规检查和批量上架模板导出，降低跨境卖家人工搬运商品的时间成本和出错率。

---

## 一、项目解决的核心痛点

跨境电商卖家在从国内货源平台搬运商品到 TikTok Shop 等海外平台时，通常会遇到以下问题：

### 1. 商品采集效率低

运营人员需要手动打开 1688 商品页面，复制商品标题、价格、规格、主图、详情图和商品描述，再整理成平台要求的格式。  
一个商品从采集到整理完成通常需要 20-30 分钟，批量铺货时效率非常低。

### 2. 中文内容不适合海外平台

1688 商品标题和详情大多是中文，并且包含大量国内电商表达方式。  
直接翻译容易出现语言生硬、关键词缺失、卖点不清晰、不符合当地消费者搜索习惯等问题。

### 3. 商品图片不符合跨境平台要求

很多 1688 商品图片包含中文文字、水印、厂家信息、资质证书、联系方式、二维码等内容。  
这些图片直接上传到 TikTok Shop 可能影响审核、转化率和用户体验。

### 4. TikTok Shop 上架字段复杂

TikTok Shop 批量上传模板字段较多，包括标题、描述、类目、SKU、规格、价格、库存、图片链接等。  
人工填写 Excel 模板容易出现字段缺失、格式错误、图片链接异常等问题。

### 5. 多店铺、多国家运营成本高

不同国家站点需要不同语言、不同标题风格、不同价格策略和不同平台合规要求。  
人工处理多个店铺和多个国家站点会消耗大量运营时间。

核心能力包括：

- 1688 商品信息采集
- 商品标题、规格、描述结构化
- 中文商品内容清洗
- 商品标题多语言翻译
- 商品卖点本地化改写
- 商品图片 OCR 检测
- 图片中文、水印、无效信息识别
- 商品主图质量评分
- 商品图片合规检查
- TikTok Shop 标题长度检查
- TikTok Shop 敏感词检查
- 商品类目建议
- 商品 SEO 关键词优化
- TikTok 官方批量上传 Excel 模板导出
- 多 Agent 任务状态流转
- Token 使用量统计与优化

## 三、系统核心逻辑

本项目不是单一 AI 问答工具，而是一个完整的长链路 Agent 工作流。

整体流程如下：

```text
商品输入
   ↓
采集 Agent
   ↓
清洗 Agent
   ↓
翻译 Agent
   ↓
图片 Agent
   ↓
合规 Agent
   ↓
导出 Agent
   ↓
TikTok Shop 批量上架 Excel
##四、六大核心 Agent 设计
1. 采集 Agent

采集 Agent 负责从商品页面或用户输入中提取基础商品信息。

主要任务：

提取商品标题
提取商品价格
提取商品规格
提取商品 SKU
提取商品主图
提取商品详情图
提取商品描述
将非结构化商品数据转化为结构化 JSON 数据

输出示例：

{
  "title": "厨房不锈钢置物架",
  "price": "36.80",
  "images": [],
  "sku": [],
  "description": ""
}
2. 清洗 Agent

清洗 Agent 负责对采集到的商品内容进行过滤和清理。

主要任务：

删除无效字段
过滤重复图片
过滤低质量图片
识别工厂资质图
识别二维码图片
识别联系方式图片
整理商品规格
提取真正适合上架的商品卖点

清洗后的数据会变成更适合 AI 翻译、图片优化和平台上架的标准数据。

3. 翻译 Agent

翻译 Agent 负责将中文商品内容转化为目标市场语言。

主要任务：

中文标题翻译为英文
商品卖点本地化改写
商品描述多语言生成
根据目标国家优化表达方式
根据跨境电商搜索习惯生成关键词
避免直译、生硬翻译和无效关键词堆砌

示例：

原始标题：
厨房不锈钢多层置物架家用落地收纳架

优化后标题：
Stainless Steel Multi-Layer Kitchen Storage Rack for Home Organization
4. 图片 Agent

图片 Agent 负责处理商品图片相关任务。

主要任务：

OCR 识别图片中文字
检测水印
检测中文营销字
检测二维码
检测厂家联系方式
检测低清晰度图片
给主图打分
推荐最佳商品主图
标记需要重做的图片
为后续生成场景图、营销图提供基础数据

图片 Agent 的作用是避免卖家直接上传不适合海外平台的 1688 原图。

5. 合规 Agent

合规 Agent 负责检查商品是否符合 TikTok Shop 等平台的基础上架要求。

主要任务：

检查标题长度
检查敏感词
检查违禁词
检查类目是否匹配
检查图片是否可能影响审核
检查商品描述是否完整
检查 SKU、价格、库存字段是否完整
输出商品合规评分

输出示例：

{
  "status": "passed",
  "score": 92,
  "warnings": [
    "Title length is close to platform limit",
    "Some detail images contain Chinese text"
  ]
}
6. 导出 Agent

导出 Agent 负责将最终处理好的商品数据生成平台可用文件。

主要任务：

生成 TikTok Shop 批量上传 Excel
按平台字段映射商品数据
自动填充标题、描述、价格、库存、SKU、图片链接
检查必填字段
输出可下载文件
为后续自动上架流程提供标准数据

最终输出：

TikTok_Shop_Product_Upload_Template.xlsx
##五、是否包含长链推理

本项目包含长链推理能力。

原因是系统不是简单地调用一次 AI 生成标题，而是需要根据商品信息、目标市场、图片质量、平台规则、SKU 信息和合规要求进行连续判断。

例如，一个商品进入系统后，AI 需要依次判断：

这个商品是否适合跨境销售
商品标题应该保留哪些核心关键词
哪些中文词需要翻译，哪些词需要本地化改写
哪些图片可以作为主图
哪些图片包含中文、水印或厂家信息
商品描述是否符合海外消费者阅读习惯
标题长度是否符合 TikTok Shop 规则
类目是否匹配
是否可以导出到官方模板
是否需要人工复核

因此，本系统属于长链路、多步骤、多判断的 AI Agent 工作流。

##六、是否包含多 Agent 协作

本项目包含多 Agent 协作。

不同 Agent 不直接混在一起处理所有任务，而是按照职责分工协作：

Agent	主要职责
采集 Agent	获取商品原始数据
清洗 Agent	过滤无效内容
翻译 Agent	多语言翻译和本地化
图片 Agent	OCR、图片检测、主图评分
合规 Agent	平台规则检查
导出 Agent	生成上架 Excel 文件

每个 Agent 接收上一个 Agent 的输出，并生成下一个 Agent 所需的数据。

这种设计的优势是：

逻辑清晰
方便扩展
方便替换不同 AI 模型
方便定位错误
方便统计每个环节的 Token 消耗
方便后续扩展为 SaaS 系统
##七、Token Plan 设计

为了降低 AI 调用成本，本项目设计了 Token 使用优化方案。

1. 轻量任务使用低成本模型

例如：

标题翻译
卖点改写
商品描述生成
类目初筛
敏感词初步检查

这些任务可以使用轻量模型完成，降低 Token 成本。

2. 复杂任务使用高能力模型

例如：

图片理解
OCR 结果分析
商品合规判断
商品卖点提炼
商品质量评分
多步骤推理决策

这些任务交给能力更强的模型处理，提高准确率。

3. 建立缓存机制

系统会对以下内容建立缓存：

相同商品链接
相同商品标题
相同图片 URL
相似商品描述
重复翻译结果
重复合规检查结果

避免同一个商品重复消耗 Token。

4. 分阶段调用 AI

不是所有商品都直接进入完整 AI 流程。

系统会先进行基础规则判断：

商品数据是否完整
是否缺少图片
是否缺少价格
是否明显不适合上架
是否为重复商品

只有通过基础检查后，才进入后续 AI 处理流程。

5. 记录 Token 消耗

系统会记录每个任务的 Token 消耗，方便后续进行成本分析。

可统计内容包括：

每个商品消耗多少 Token
每个 Agent 消耗多少 Token
每个用户消耗多少 Token
每个店铺消耗多少 Token
哪类商品 AI 成本最高
哪类任务最消耗 Token

##十一、项目运行方式
1. 克隆项目
git clone https://github.com/your-name/bysheen-agent-system.git
cd bysheen-agent-system
2. 配置环境变量
cp .env.example .env

在 .env 中填写 AI 接口配置：

OPENAI_API_KEY=your_api_key
OPENAI_BASE_URL=https://api.openai.com/v1
MODEL_NAME=gpt-4o-mini
3. 启动项目
docker compose up --build
4. 访问系统
前端地址：http://localhost:5173
后端地址：http://localhost:8000
接口文档：http://localhost:8000/docs
